ブラウザ、Node.js、テスト環境。
「複数環境を扱うとき、使えないAPIや型を補完してしまったことはありませんか？」
もし作成中のプロジェクトに `tsconfig.json` が1つしか存在しないのであれば、その設定は間違っています。

:::message alert
Vite や Next.js などの公式が配っている初期ひな形は手軽に始められる反面、設定が意図的に簡素化されています。
厳しい表現をすると、これらの初期ひな形は型に対してルーズです。初期のセットアップを簡単にするため、本来は間違っている設定を提供してしまっています。これこそが、型が互いに混ざり合って汚れてしまう「型汚染」の根本原因です。
:::

この記事では、まず現代のWeb開発における複数環境の実態を整理します。実務で遭遇して混乱した体験談を交えながら、中〜大規模なプロジェクトで実践すべき設定ファイルの分割方針を解説します。無用な混乱を防ぐ**「正しい設定ファイルの作り方」**についても触れます。

## 前提：現代の開発で「1つの環境しかない」プロジェクトは存在しない

かつてのように「1つの HTML と 1つの JavaScript ファイルを書き、そのままブラウザに読み込ませて動かす」という素朴な開発スタイルであれば、環境は「ブラウザ」の1つだけでした。

しかし、ツールによるビルドや検証が当たり前になっている現代において、単一の環境だけで完結するプロジェクトは存在しないに等しいです。

たとえ画面を作るだけのシンプルなプロジェクトであっても、裏側では最低でも以下の3つの異なる環境が同じリポジトリ内で同時に動いています。

1. **ブラウザ環境**（実際にユーザーが触る本番コード）
   `window` や `document` などの DOM API が存在する環境。
2. **Node.js環境**（開発環境・ビルドツールなど）
   ViteやNext.jsの設定ファイル、LinterやFormatterの設定ファイル、自作の自動化スクリプトが動く環境。`process.env` や `fs`, `path` といった Node.js 専用の機能が存在する。
3. **テスト実行環境**（テストコード）
   Node.js 上でテストツールを動かし、`jsdom` や `happy-dom` でブラウザの動きを再現する複合的な環境。

:::message
💡 注記
実機ブラウザや独自ブラウザエンジン上で直接テストを動かす「Vitest browser mode」のみを使ってテスト環境を構築している場合は、この限りではありません。しかし、プロジェクト全体のテストをそのような構成のみで完結させているケースは非常に稀です。
:::

このように、1つのプロジェクトには実行環境の性質がまったく異なるコード群を混在させています。設定ファイルを1つにまとめてしまうと、異なる環境の型定義がすべて混ざり合い、お互いを壊し合うことになります。

## 具体的な課題：tsconfigが1つしかないプロジェクトで発生する問題

型が混ざり合うと、どのような不都合が起きるのか。実務で実際に遭遇した2つの事例を紹介します。

### 課題1：setTimeoutの戻り値がNode.jsの型になってしまう

ブラウザ向けのアプリケーションコードを書いていたとき、タイマー処理を実装するために `setTimeout` を使いました。

```ts:src/main.ts
const timerId = setTimeout(() => {
  console.log('Hello');
}, 1000);
```

このとき、`timerId` の型をエディタで確認すると、戻り値の型が `number` ではなく、`NodeJS.Timeout` になっていました。

本来、ブラウザが提供する `setTimeout` は数値（`number`）を返しますが、Node.js の `setTimeout` はオブジェクト（`NodeJS.Timeout`）を返します。

原因は、プロジェクトの `tsconfig.json` で Node.js の型定義（`@types/node`）が全体に読み込まれていることです。ブラウザの型定義が Node.js のもので上書きされて混ざってしまったのです。

プロジェクトに参加したばかりのメンバーや、Webの仕組みを学び始めたばかりの人にとって、この挙動は不親切極まりありません。「ブラウザのコードなのに、なぜ Node.js の型が返ってくるのか」という矛盾は、仕組みの正しい理解を妨げ、大きな誤解を生む原因になります。

### 課題2：本来動かないはずのAPIが補完され、実際に動かしたときに落ちる

逆に、Node.jsで動くはずの設定ファイルやビルド用のスクリプトを書いているときに、ブラウザ専用のAPIを入力してしまい、警告が出ないまま使えてしまうことがあります。

たとえば、Node.js向けの設定ファイルで `lib: ["DOM"]` が有効になっているとします。`document` や `window` など、本来そのスクリプトの実行環境（Node.js）では使えないAPIまで補完されてしまいます。

```ts:vite.config.ts（Node.js環境）
const width = document.body.clientWidth;
```

このコードは、型チェックの段階ではエラーになりません。しかし、実際にビルドツールを動かした（実行した）瞬間に「document is not defined」とエラーになり、プログラムが強制終了します。

:::message alert
特にグローバルに存在するオブジェクトや関数は、このような型汚染の影響を強く受けます。
本来その環境に存在しないものが使えてしまう状態は、深刻な不具合を引き起こす原因になります。開発者が正しい知識に基づいて、その環境に存在する正しいAPIだけを操作できる状態を作ることが必要です。
:::

## 解決策1：環境（ランタイム）ごとにベースとなるtsconfigを作る

この問題を解決する根本的なやり方は、**「コードが動く環境ごとに設定ファイルを作成し、それぞれの関係性を適切に繋ぐ機能（Project References）で束ねる」**ことです。

まずは、コードが動作する純粋な実行環境（ランタイム）ごとに、基本となる設定ファイルを用意します。ここでは名前を環境名だけで統一します。

- `tsconfig.browser.json` : ブラウザ用のコード向けの設定。`lib` に `"DOM"` などを指定し、Node.js の型定義（`@types/node`）は確実に除外する。
- `tsconfig.node.json` : 開発環境のスクリプトや設定ファイル向けの設定。Node.js の型定義を含める。
- `tsconfig.jsdom.json` : テスト時のNode.js環境で、ブラウザAPIを擬似的に動かせるようにした設定。

そして、プロジェクトルートにある `tsconfig.json` は、これらを参照するだけの役割（すべての設定を束ねる起点）にします。

ここで重要なのは TypeScript のSolution設定の作法を守ることです。ルートの設定ファイルには `"files": []` を置き、参照される側の設定ファイルには `"composite": true` を指定します。

```json:tsconfig.json
{
  "files": [],
  "references": [
    { "path": "./tsconfig.browser.json" },
    { "path": "./tsconfig.node.json" },
    { "path": "./tsconfig.jsdom.json" }
  ]
}
```

これで、エディタは「いま開いているファイルがどの環境向けか」を正確に理解します。ブラウザ用のコードでは `number` 型を、Node.js用のコードでは `NodeJS.Timeout` を正しく提示するようになります。

## 解決策2：用途ごとにtsconfigを作り、環境用tsconfigを継承（extends）する

環境ベースの設定ができたら、次は実際にコマンドを実行する「用途（アプリケーション開発、ビルド、テストなど）」に合わせた設定ファイルを作ります。

ここでは命名規則を用途名で統一し、先ほど作成した「環境ベース」の設定を `extends`（継承）して組み合わせます。

- `tsconfig.app.json` : 本番アプリケーションの開発・ビルド用。`tsconfig.browser.json` を継承する
- `tsconfig.dev.json` : 設定ファイルやビルド用スクリプトの実行用。`tsconfig.node.json` を継承する
- `tsconfig.test.json` : テストコードの実行用。`tsconfig.jsdom.json` を継承する

Project References の対象となるこれらの設定ファイルには、ビルドの依存関係を制御するための `"composite": true` を必ず含める必要があります。

```json:tsconfig.app.json
{
  "extends": "./tsconfig.browser.json",
  "compilerOptions": {
    "composite": true,
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "include": ["src"]
}
```

この「環境」と「用途」を分ける設計は、複数のプロジェクトを1つにまとめるモノレポ構成においても力を発揮します。
共通の環境ベース設定をルートディレクトリに置き、各個別プロジェクトの用途別設定ファイルからそれを継承するだけで、高い安全性を保ったままシンプルな設定ファイルを量産できます。

## 解決策3：環境ベースのtsconfigはコミュニティの推奨設定（@tsconfig/\*）を利用する

環境と用途を分けて設計する重要性を理解できても、複雑なTypeScriptの設定をゼロから書き上げるのは難しいです。

そこで、`@tsconfig/*` という名前空間で公開されている、コミュニティ管理の推奨ベース設定を利用することをおすすめします。

https://www.npmjs.com/org/tsconfig

`@tsconfig/*` では、Node.jsの各バージョン向けや、ブラウザ（DOM）向けなど、厳密に管理された環境ごとの推奨設定ファイルが公開されています。これを利用すれば、細かい設定項目の意味をすべて理解していなくても、広く使われているベース設定を出発点にできるため、安全な設定をすぐに取り入れることができます。

### 具体的な運用のやり方

**1. パッケージをインストールして継承する**

対象の環境パッケージ（例：`@tsconfig/node18` や厳格なチェックを行う `@tsconfig/strictest` など）をインストールし、先ほど設計した環境用設定ファイルで継承します。

```json:tsconfig.node.json
{
  "extends": "@tsconfig/node18/tsconfig.json",
  "compilerOptions": {
    "composite": true
  },
  "include": ["scripts", "*.config.ts"]
}
```

**2. 外部パッケージを増やしたくない場合は、コピーしてプロジェクト内に直接配置する**

管理するパッケージを増やしたくない場合は、`@tsconfig/*` の公式リポジトリから設定内容（JSON）をコピーしましょう。プロジェクト内に `tsconfig.base.json` などの名前で直接配置し、それを継承するやり方がおすすめです。

:::details 💡 なぜ直接書き込まずに継承（extends）を使うのか？
コピーした設定を、そのまま自分たちの `tsconfig.json` にすべて貼り付けても動きます。しかし、それをやってしまうと**「どこまでが標準の推奨設定で、どこからが自分たちで独自に上書きした設定なのか」が分からなくなってしまいます**。

継承機能を使ってベースの設定と明確に分離しておけば、独自に変更した箇所だけが手元に残ります。プロジェクト固有の「変更した意図」が明確に伝わり、きれいな設定ファイルを維持できます。
:::

## まとめ

公式の初期ひな形は、手軽に動かすことを最優先にして設定を省略しています。そのまま使い続けると型定義が混ざり合い、問題が発生します。

TypeScript が持つ型安全の恩恵を最大限に受け、開発者が迷いなく正しいコードを書けるようにするためには、以下の2つの手順が重要です。

1. **実行環境（ランタイム）ごとに設定ファイルを分け、用途別ファイルから継承して組み合わせる。**
2. **ベースとなる環境設定は `@tsconfig/*` などのコミュニティ推奨設定を利用し、独自の追加設定と明確に分ける。**

現在、1つの `tsconfig.json` にすべての設定を詰め込んでいるプロジェクトがあれば、ぜひこの機会に分割を試してみてください。
